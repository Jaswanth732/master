Create Batch (createBatch)

Purpose

To provide the consumer with the ability to create a batch in a single call.

Service Flow Description

The Create Batch service logic is

  • An API Consumer invokes a service providing a Batch Object

  • Validate Batch Object Structure: The Batch object is validated if it is schematically correct
    • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message :

    Error Code

    Message
    APICLM.BUS.ERROR.00001 Invalid object - Unsupported message type
    APICLM.BUS.ERROR.00010 Batch amount currency Id is missing
  • Payment Date Exist and Valid - Yes : If payment Date is provided (eg non-empty) and is greater than Received Date then it will be used as the Batch payment date

  • Payment Date Exist and Valid - No : If payment Date is not provided or is less than the Received date, then service BatchBAS.calculatePaymentDate will be called to calculate it
  • Create Batch: The purpose of this step is to persist the Batch object in MedNeXt+DB
    • If Batch object is not persisted due to service exception, an HTTP 200 response will be propagated to the caller containing an Error Information message
  • Create Batch Case : The Batch business process (ProcessBatchCase) will be initiated . A new "Create Batch Human Task" will be created and assigned to the least busy user of "Batch Creator" role.
    • The Case initiation will be performed asynchronously and will not block the actual service response
  • The service will reply with the created Batch object enriched by the new batch ID

Purpose

To retrieve a list of batches with their related information based on specific criteria.

Prerequisites

  • At least one of the above search criteria must be provided in the service request to restrict the returned results. When no search criteria are provided, the service will respond back an error requesting at least one of the search criteria to be provided.

    • submittedBy : Provider or Insurance Company
    • externalReferenceId : Batch External Reference
    • pageNumber : pagination result set number

Adjust Claim Reserve (adjustClaimReserve)

Purpose

To provide the consumer with the ability to update a Claim Reserve in MedNeXt in a single call.

Service Description

  • Provide only the attributes of an existing claim reserve which he/she wishes to update.
  • Provide only the attributes of existing claim reserve sub-entities (entity unique ids included - e.g. Reserve Detail Number for Reserve Details) he/she wishes to update.
  • Provide new sub-entities he/she would like to create under an existing Claim Reserve (e.g. new Reserve details) Unsupported Updates.

Process Logic

    The Adjust Claim Reserve service logic is:

  • An API Consumer invokes a service providing the information of the claim reserve to be adjusted.
  • Validation of Request Message Structure: The Request object will be validated if it is schematically correct. If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message.
  • After that, the consumer will be informed with a queue number. The Queue Number will be created to handle the asynchronous Adjust Claim Reserve process and will represent the whole request and will contain a list of queue items, one for each claim to be adjusted. If the call fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
  • In the meanwhile, a new process will be initiated to make additional validations. These validations will check if the claim reserve has no pending tasks, the claim number exists in status Open and the information provided from the consumer is businesswised correct:
    • Claim Number and Reserve Level Indicator field is not allowed to be modified.
    • Total Reserve Amount must be greater than or equal to zero.
    • When Reserve Number value is provided, it will be used by the system in combination with the Claim Number to identify the reserve entry to be updated. It should be validated against valid reserve numbers of the claim. When no Reserve Number value provided, a new reserve record will be created (Reserve Number will be assigned by the system) with the provided Claim Reserve Information for the provided claim.
    • Reserve Date is not allowed to be updated when there is Reserve Type Id='REC' and the Reserve Date provided is earlier than the Reserve Date of the reconciliation record. Otherwise: Reserve Date must be less than or equal to the system current date Reserve Date must be greater than or equal to the claim occurrence date Reserve date must be greater or equal to the latest Reconciliation date
    • Reserve Amount is not allowed to be updated when there is Reserve Type Id='REC' and the Reserve Date provided is earlier than the Reserve Date of the reconciliation record. Otherwise: If Reserve Level Indicator=''Claim'' then the field is mandatory and must be numeric. In any other case it will be ignored.
    • Reserve Type Id is not allowed to be updated when there is Reserve Type='REC' and the Reserve Date provided is earlier than the Reserve Date of the reconciliation record. Otherwise It must be a valid value based on the provided: 'INIT'-Initial 'ADD'-Additional
    • Total Reserve Detail Amount is applicable only If Reserve Level Indicator=''Package'' or ''Plan'' or ''Benefit''
    • Reserve Details must be provided if Reserve Level Indicator=''Package'' or ''Plan'' or ''Benefit'' then the Reserve Detail, otherwise will be ignored.
    • Reserve Details Level value must be the package number of the provided member of the claim if Reserve Level Indicator=''Package''. If Reserve Level Indicator=''Plan'', then the Reserve Details Level value provided must be the plan number which belongs to the package of the provided member of the claim. If Reserve Level Indicator=''Benefit'', then the Reserve Details Level value provided must be the benefit number which belongs to the package of the provided member of the claim.
    • Reserve Details Amount is not allowed to be updated when there is Reserve Type Id="REC" and the Reserve Date provided is earlier than the Reserve Date of the reconciliation record.
  • Reply: The service will reply with HTTP 202 - Accepted along with the created Queue Item object

  • An API Consumer invokes a service providing at least one invoice number

    • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an message


  • Create Queue Item for each invoice number : A new Queue Item will be created to handle the asynchronous cancel invoice process. Service QueueAPIServices.create will be used and will reply with a Queue Item. Each queue item will be consumed by ClaimAPIServices.PerformCancelInvoiceBPS
    • If the call fails, an If the call fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

  • A database poller will retrieve each created queue item and initiate a backend service (ClaimAPIServices.PerformCancelInvoiceBPS) to check if the invoice is eligible for cancellation and create a Cancel Invoice Task

  • Reply :The service will reply with the list of the queue items (One queue item per invoice).


Asynchronous Processing

An asynchronous service will process each invoice performing the following basic steps :

    • Check Invoice Cancellation Eligibility : A set of eligibility checks are performed over the invoice
      • The invoice is eligible for cancellation if it meets the below conditions :
        • Invoice status is reported and has not been previously corrected or cancelled and
          • There is no active resubmit invoice task linked to this invoice
          • There is no active correct invoice task linked to this invoice.
          • There is no active cancel invoice task linked to this invoice.
          • The invoice is not a negative invoice resulted from a previous adjustment process
        • Invoice status is calculated and valid and has not been previously corrected or cancelled and
          • the invoice is not linked to an active review task (technical review, medical review etc).
          • invoice is not linked to an active correct invoice task
          • invoice is not linked to an active resubmit invoice task.
          • Invoice is not linked to a cancel invoice task.
          • The invoice is not negative
          • If the invoice is a pre-invoice it should not be linked to an invoice

    • Is invoice eligible for cancellation - Yes :
      • Cancel Invoice : A service is called to cancel the invoice
      • If Invoice Original Status - Calculated & Valid :
        • Update existing running business process: The existing business process will be updated and milestone "Invoice Cancelled" will be achieved
        • Update Queue Item : The Queue item will be updated with the following details:
          • Status : COMPLETED
          • Resource Id : The cancelled invoice number
          • Info Code : CLM.BUS.INFO.40051
          • Info Message : Invoice {Invoice Number} successfully cancelled.
      • If Invoice Original Status - Reported:
        • Create new business process : A new business process will be initiated and milestone "Invoice Cancelled" will be achieved
        • Updated Queue Item : The Queue Item will be updated with the following details:
          • Status : COMPLETED
          • Resource Id : The newly created negative invoice as a result of the cancellation action
          • Info Code : CLM.BUS.INFO.40052
          • Info Message : Invoice {invoice Number} successfully cancelled. Negative invoice {negative Invoice Number} has been created.
    • Is invoice eligible for cancellation - No :
      • Update Queue Item : The Queue Item will be updated with the following details:
        • Status : ERROR
        • Error Code : APICLM.BUS.ERROR.00030
        • Error Message: Invoice is not eligible for cancellation

Correct Invoice (correctInvoice)

Purpose

To provide the user with the ability to correct an existing invoice in MedNeXt.

Service Flow Description

Unsupported Updates

The following corrections on an existing invoice will not be supported by the Correct Invoice Service

  • Changes on the invoice sub-entity ids meaning that :
    • If a sub-entity (eg. a Note) is provided with a Note ID which does not exist, the provided Note ID will be ignored and a new Note will be created in MedNeXt+ DB. A new Note ID wil be generated.
    • Deletion of sub-entities: The absence of sub-entities from the service input, will not lead to their deletion.

  • Updates on one of the following ids will not be allowed. If the following values sent by the API Consumer differ from the ones stored in MedNeXt+ Database then the service will reply with an error

    Entity Read Only attribute
    Invoice Invoice Number
    Invoice Exchange Rate Exchange Rate Number
    Invoice Note Note Number
    Invoice Line Line Number
    Incident Incident Number
    Incident Diagnosis Diagnosis Number
    Case Case Number
    Claim Claim Number
    Custom Fields Field Number
    Invoice Line Amount Amount Number

The Correct Invoice service logic is

  • An API Consumer invokes a service providing an Invoice Object
  • Validate Invoice Message Structure: The Invoice object will be validated if it is schemantically correct
    • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

  • Retrieve Invoice From MedNeXt+ DB: Using the invoice number provided by the Invoice input Object , the Invoice object is retrieved from MedNeXt+DB

    • If the Invoice entity is not found in the database an HTTP 404 - Not Found error will be propagated to the caller along with an Error Information message


  • Business Validations
    • Check invoice eligibility for correction : This step checks if this invoice is eligible for update. It performs the following checks :
      • Invoice Status is Reported or Calculated And Valid
      • Invoice has not previously been Corrected or Cancelled
      • There is no active Resubmit, Correct,Cancel invoice task
      • The invoice is not a negative or a pre-invoice

      • If the invoice is found as non eligible for correction an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

        Error Code

        Message
        APICLM.BUS.ERROR.00029 Invoice is not eligible for correction



    • Merge Input with DB Retrieved Invoice : The input provided invoice instance is validated against the retrieved invoice instance. If the updates are valid then these two invoice images will be merged into one. The steps to do so are
      • Validate Requested Updates : A series of validation checks are performed to determine if an unsupported update is requested.
        • If at least one unsupported update is found an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
      • Retrieve Batch(if exists) : If batch number is provided in the service input then the batch Object will be retrieved
      • Perform Invoice Business Validations:
        • The merged Invoice Object along with the retrieved Batch Object (if exists) will be validated against a set of business rules> Validations section , (Entity =Invoice) which check the data validity.
          • If the Validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

      • Perform Invoice Currency Validations
        • Retrieve Invoice supported Payment Currencies : A service will be used to retrieve the payment restricted list of currencies which are configured at system level. The service will also take into account the customer configured preference "Additional Payment Currencies". (service CurrencyDS.queryInvoicePaymentCurrencies )

      • Check Invoice Payment Currency: Validate if the provided invoice payment currency is within the supported Invoice Payment Currency list.

        • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message


          Error Code

          Message
          APICLM.BUS.ERROR.00025 Not supported Invoice Payment Currency.





    • Correct Invoice :
      • Invoice Status = Calculated & Valid or Reported: When the invoice is in one of these two statuses it cannot be directly updated. The original invoice must be adjusted meaning a new invoice will be created as a copy of the original invoice and will be updated according to the input invoice image. The steps are :
        • Adjust Invoice : The original invoice will be Cancelled and a new Invoice will be created as a copy of the original one
          • If the Adjustment action fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
        • Update Adjusted Invoice : The Adjusted Invoice image will be merged with the requested updates. The merged image will be persisted in MedNeXt+ DB
        • Create Invoice Link : The link between the original and the new invoice (as created by the adjustment step) will be saved to trace the results of the adjustment action. The type of the link will be "CORRECT INVOICE"
        • Copy Bypass Line Transactions : Any bypass decisions taken over raised inconsistencies upon the original Invoice's Lines must be copied to the new invoice.This will prevent the computation process to raise the same inconsistencies when executed for the new invoice

      • Invoice Status = New or Pending : When an invoice is in either of these two statuses it means it can be updated
        • Update Invoice : The existing invoice is updated (
          • If the Update Invoice action fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
        • Copy Bypass Line Transactions : Any bypass decisions taken over raised inconcistencies upon the original Invoice Lines must be copied over the new invoice.
          • If the Copy Bypass Transactions action fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

    • Start Invoice Process : In case the Invoice creation is successful then a ProcessInvoiceCase Process will be triggered. As ProcessInvoiceCase support multiple starting points, it must start from the point where Auto Review Process
      • Auto review process will try to Compute the Invoice if a list of Business Rules are evaluated as true (See Validations section , Entity =Auto Review). The set of rules are evaluated by the AutoReview process and not the API itself.
        • If the at least one Business Rule evaluates to false, Auto Review process will not be executed and A Correct Invoice HumanTask will be created for manually handling the invoice
        • If all Business Rules evaluate to true, then Auto Review process will be executed and the invoice will be computed. Depending on the computation results:
          • A Review Inconsistencies (Technical or Medical) Human tasks will be created if the computation process, raised inconsistencies
          • The invoice will be marked as "Ready for reporting" if the computation process finished successfully

      Information

      The "Start Invoice Process" describes the logic of ProcessInvoiceCase Process. API will only initiate ProcessInvoiceCase and will proceed to next step (Reply)

  • Reply :The service will reply with the corrected invoice object enriched with the new invoice ID

Create Invoice (createInvoice)

Purpose

To provide the consumer with the ability to create an invoice and its related information (e.g. diagnosis, lines, procedures, notes, etc) in MedNeXt in a single call.

Service Flow Description

The Create Invoice service logic is

  • An API Consumer invokes a service providing an Invoice Object
  • Validate Invoice Message Structure: The Invoice object will be validated if it is schemantically correct
    • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message


  • Retrieve Batch(if exists) : If batch number is provided in the service input then the batch Object will be retrieved
  • Perform Invoice Business Validations: The received Invoice Object will be validated against a set of business rules(See Validations section , Entity =Invoice) which check the data validity .
    • If the Validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

  • Perform Invoice Currency Validations
    • Retrieve Invoice supported Payment Currencies : A service will be used to retrieve the payment restricted list of currencies which are configured at system level. The service will also take into account the customer configured preference "Additional Payment Currencies".

    • Check Invoice Payment Currency : Validate if the provided invoice payment currency is within the supported Invoice Payment Currency list.

      • If the validation fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

        Error Code

        Message
        APICLM.BUS.ERROR.00025 Not supported Invoice Payment Currency.
  • Create Invoice : The purpose of this process step is to persist the Invoice object(and its child entities such as Invoice Lines, Incident, Exchange Rates etc) in MedNeXt+DB. The steps to do so are :
    • Incident Checks:
      • If an incident number is provided then, it will be retrieved from MedNeXt+ DB using IncidentBAS.queryIncidentById service .
        • If the Incident is not found an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
        • If the Incident exists, then the invoice will be linked with this incident

      • If an incident object without an incident number is provided then the existing API Incident service (POST request on /mednext/authorisation/incident) will be called to create the incident and the Case subentity.
        • If the Incident service fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message

    • Claims checks :
      • If an claim number is provided then, it will be retrieved from MedNeXt+ DB
        • If the Claim is not found an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
        • If the Claim exists, then the invoice will be linked with this Claim

      • If an Claim object without a Claim number is provided then the existing API Claim service (POST request on /mednext/claim/claim) will be called to create the Claim entiy.
        • If the Claim service fails, an HTTP 400 - Bad request error will be propagated to the caller along with an Error Information message
    • A Service will be used to persist Invoice and every sub-entity.
      • The Incident will be automatically linked with the Invoice
      • The case will be automatically linked with the Incident
        • If the Invoice is not persisted due to service exception, an HTTP 400 response will be propagated to the caller containing an Error Information message

  • Start Invoice Process : In case the Invoice creation is succesful then a ProcessInvoiceCase Process will be triggered. As ProcessInvoiceCase support multiple starting points, it must start from the point of Auto Review Process :
    • Auto review process will try to Compute the Invoice if a list of Business Rules are evaluated as true (See Validations section , Entity =Auto Review). The set of rules are evaluated by the AutoReview process and not the API itself.
      • If the at least one Business Rule evaluates to false, Auto Review process will not be executed and A Correct Invoice HumanTask will be created for manually handling the invoice
      • If all Business Rules evaluate to true, then Auto Review process will be executed and the invoice will be computed. Depending on the computation results:
        • A Review Inconsistencies (Technical or Medical) Human tasks will be created if the computation process raised inconsistencies
        • The invoice will be marked as "Ready for reporting" if the computation process finished successfully

    Information

    The "Start Invoice Process" describes the logic of ProcessInvoiceCase Process. API will only initiate ProcessInvoiceCase and will proceed to next step (Reply)

  • Reply :The service will reply with the created invoice object enriched with the new invoice ID
